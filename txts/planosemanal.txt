ou assumir uma versão de projeto com:
	•	Python
	•	Streamlit
	•	NumPy / Pandas
	•	Matplotlib ou Plotly
	•	SQLite ou JSON
	•	modelo aerodinâmico simplificado

Plano de desenvolvimento do projeto

Semana 1 — Definir bem o projeto

Nesta fase o objetivo é deixares de ter só uma ideia e passares a ter um projeto com fronteiras claras.

O que fazes
	•	escolher o nome do projeto
	•	escrever uma descrição curta
	•	decidir exatamente o que vais simular
	•	decidir quais são os parâmetros de entrada
	•	decidir quais são os resultados de saída
	•	definir simplificações

Tens de sair desta semana com isto resolvido
	•	o projeto vai focar-se numa asa simplificada
	•	inputs: ângulo de ataque, velocidade, área, curvatura, altura ao solo se quiseres
	•	outputs: drag, downforce, eficiência aerodinâmica
	•	sem CFD real
	•	sem carro completo
	•	sem regulamentos complexos

Entrega da semana

Um ficheiro tipo README.md com:
	•	nome do projeto
	•	objetivo
	•	funcionalidades
	•	tecnologias
	•	limitações

⸻

Aqui montas o ambiente e a estrutura do projeto.

O que fazes
	•	criar a pasta do projeto
	•	criar ambiente virtual

	•	criar primeira versão da aplicação

Semana 3 — Criar o modelo da asa

Agora defines como vais representar a tua asa no código.

O que fazes
	•	criar uma classe ou estrutura para a asa
	•	guardar parâmetros
	•	validar valores
	•	preparar funções auxiliares

Exemplo de parâmetros
	•	angle_of_attack
	•	velocity
	•	area
	•	camber
	•	air_density

Entrega da semana

Uma classe tipo WingConfig ou semelhante, já funcional.

⸻

Semana 4 — Implementar o motor de cálculo simplificado

Aqui começas a dar “vida” ao projeto.

O que fazes
	•	implementar fórmulas simplificadas
	•	calcular drag
	•	calcular downforce
	•	calcular eficiência

Objetivo

Não interessa que seja um modelo hiper-realista. Interessa que:
	•	faça sentido
	•	responda logicamente às alterações
	•	seja consistente

Por exemplo:
	•	se aumentas ângulo de ataque, downforce sobe até certo ponto
	•	drag também sobe
	•	a eficiência muda consoante a combinação

Entrega da semana

Um script que recebe parâmetros e devolve métricas calculadas.

⸻

Semana 5 — Fazer testes básicos ao modelo

Nesta fase validas se o sistema pelo menos se comporta de forma lógica.

O que fazes
	•	testar vários valores manualmente
	•	ver se os resultados variam como esperado
	•	corrigir absurdos
	•	definir limites mínimos e máximos

Exemplos de testes
	•	velocidade baixa vs alta
	•	ângulo pequeno vs grande
	•	área pequena vs grande

Entrega da semana

Uma pequena tabela de testes e observações.

Semana 10 — Implementar otimização simples

Aqui entra a parte mais interessante de engenharia informática.

O que fazes
	•	criar um algoritmo de procura
	•	definir intervalos para os parâmetros
	•	testar combinações
	•	guardar as melhores

Melhor abordagem para começar

Grid Search

Exemplo:
	•	ângulo: de 0 a 20
	•	velocidade: intervalo fixo ou dado pelo utilizador
	•	curvatura: vários níveis
	•	área: vários níveis

Depois o sistema testa tudo e devolve:
	•	top 10 por downforce
	•	top 10 por menor drag
	•	top 10 por melhor equilíbrio

Entrega da semana

Botão “Otimizar” com resultados automáticos.

Ordem prática das prioridades

Se te perderes, segue esta ordem:
	1.	modelo da asa
	2.	cálculo simplificado
	3.	interface básica
	4.	gráficos
	5.	guardar resultados
	6.	comparação
	7.	otimização
	8.	polimento

Se fizeres isto por esta ordem, nunca ficas com um projeto “bonito mas vazio”.

⸻

Versão ainda mais realista

Se achares que isto tudo é demasiado, corta o projeto em MVP + extensões.

MVP

O mínimo que tens mesmo de fazer:
	•	definir parâmetros da asa
	•	calcular drag e downforce
	•	mostrar resultados
	•	ter interface básica
	•	ter pelo menos uma forma simples de comparação

Extensões

Se houver tempo:
	•	histórico
	•	otimização automática
	•	exportação
	•	relatórios
	•	mais parâmetros
	•	mais gráficos

Isto é importante porque sozinho tens de proteger o projeto contra excesso de ambição.

⸻

Sugestão de calendário realista

Se tiveres aulas e outras cenas ao mesmo tempo, eu faria assim:
	•	Semanas 1 a 4: base e cálculo
	•	Semanas 5 a 8: interface e visualização
	•	Semanas 9 a 11: otimização
	•	Semanas 12 a 15: melhorias, documentação e apresentação
